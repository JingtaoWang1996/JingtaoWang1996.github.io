* 基于规则进行新IP地址生成。
* Entropy/IP: 香农熵研究种子地址结构，贝叶斯模型生成新的地址集合。
* [6Gen](MURDOCK A, LI F, BRAMSEN P, et al. Target generation for
  internet-wide IPv6 scanning[C]// Internet Measurement Conference,
  Nov 1-3, 2017, London, UK. New York: ACM Press,
  2017.)：基于密度聚类
* Paper2：**种子地址不平衡的问题**
* [6Tree]():基于分类层次聚类的6Tree算法，相比于6Gen，在对种子地址进行空间建模部分有线性时间复杂度。

## BACKGROUND

* IPv6 标准表示法：用32 个半字节表示,比如：**2001:0db8:85a3:08d3:1319:8a2e:0370:7344**
  * 一个字节是8位：32*4 = 128 
* IPv6 缩略表示法：对于连续为0 的部分可以压缩，使用::表示，比如**2001:0DB8::1428:57ab**
* **一个IPv6地址 = 路由前缀+子网ID+接口标识**
  * IPv6地址：全球单播+任播+多播
  * 单播地址是大规模IPv6探测的主要目标 ，因为**全球路由可达**

### 接口标识-Interface Identifier-IID

* 网络前缀为前64位

* 接口标识为后64位，用于标识链路上的接口，每条链路上接口ID必须唯一

  * **用于在同一个IPv6子网内唯一标识一个网络接口。**
  * **前缀决定你在哪个网段，IID决定你是该网段中的哪个设备接口，一个设备可能有多个网络接口**
    * **一台设备的多个网络接口：每一个网卡都可能是一个网络接口，都有一个独立的IPv6地址**
      * **因此一台设备可以有多个IPv6地址**
      * **一个接口也可以有多个IPv6地址，这就是别名。这些多个地址都可以收发流量，但逻辑上都绑定到同一个接口**

* 接口不完全等同于物理网卡，而是一个可以绑定IP地址的网络端点。

  ```
  接口 eth0:
      - Link-local: fe80::1
      - Global: 2406:2d40:3000:b724::1234
      - Temporary: 2406:2d40:3000:b724:abcd:ef01
  ```

  

* 在无状态配置Stateless Address Autoconfiguration（在没有DHCP服务器统一分配的情况下，自动生成合法的IPv6全球单播地址及相关配置参数）中，主要这是IPv6的后64位取值【包含人为决定的因素】

  * 放在Starlink的user router中，后64位除了最后一位一定是1之外，其他位基本为0
  * IPv6前64位的大部分地址都可以由公开BGP和分配地址获取

* 常见IID分配规律

  * Random：随机设定取值
  * Low-Byte：连续零值和低位零值【类型清华那篇的Starlink user router，最后的很多位都是0】
  * IPv4嵌入：在IPv6地址的最后嵌入IPv4地址，eg:`2001:db8:122:344::192.0.2.33`
  * 嵌入端口：在IPv6地址的尾部嵌入端口，eg：`2001:db8::21`
  * 嵌入英文词汇
  * 嵌入MAC地址
  * 嵌入组织唯一标识符

* 经过文中抓取一个月CERTNET北京节点的数据分析，绝大多数是随机类型地址

## Pattern-based 地址探测技术

通过分析IPv6中易被猜测的规则来生成新地址，Pattern based [《On Reconnaissance with IPv6：A Pattern-Based Scanning Approach》](chrome-extension://efaidnbmnnnibpcajpcglclefindmkaj/https://www.johannaullrich.eu/assets/papers/ullrich2015_ares.pdf)

* 人为设置一定的规则在/64地址空间进行搜索【是否可以扩展到/56或者/40 】
* 优势：避免人为分析缺陷；从作者实验结果来看这项技术发现的地址数量高于暴力扫描
* 缺点：若种子集随机化比较严重，Pattern挖掘结果不够准确，就会降低命中率【星链user router可能不存在这个问题】

## Entropy/IP 地址探测技术【貌似不太适合XL网段明确的情况】

结合信息论的香农熵、机器学习、概率图模型，给出了可视化IPv6地址结构的多种方案。主要步骤如下：

* step1-香农熵：结局对信息的量化度量问题，取值范围[0,1]，熵值越大表示该变量不确定性越大，随机性越强。
  * 计算IPv6地址集合的每半个字节的熵值（4位）*32 = 128 
* step2-地址分段：将熵值相近的半字节合并成段【以星链为例，/40的前40位无需考虑】，相近的阈值不能设置过大或过小，不然作为输入会影响后面贝叶斯建模的效率
  * PS：按照这个逻辑，实**际输入的种子地址的前40位不能差距太大**-->**确实存在种子地址不平均的情况。**
* step3-分段信息挖掘: 对步骤2划分的段做一些统计分析，包括该段最多的取值、取值最密集的范围、最大最小值，将这些信息作为贝叶斯建模的输入。
* step4-贝叶斯网络建模：根据段与段之间的取值条件概率构建整个地址的贝叶斯模型【这个图模型描述的是**每个段的取值**以及**每个段取值与之前段的条件概率**】
  * 该方式可以从贝叶斯网络图随机遍历整个贝叶斯概率模型描述的取值空间
  * Information Theory+Bayes Model in IPv6 detection
  * 可视化地址集合的熵和概率图
  * 100万种子地址耗时：112s、
  * 缺点：算法生成命中率低，对于后64位的熵值无限接近1

## 6Gen：聚类算法应用与IPv6地址发现，时间效率过低

从种子地址集合识别地址取值空间中密集区域，并在这些区域中选择新的地址取值，进而生成需要扫描的候选地址

* **基本假设：已有种子地址在不同半字节取值最密集的区域也是未知地址最密集的区域**
  * 实际分配有一定的规律：Suppose星链给用户的user router不是随机分配的，同一个组织或网络管理者可能给这一片区域分配相同的策略
* 核心步骤：DBSCAN聚类算法
  * 将每个种子地址初始作为一个聚类，在迭代过程中不断扩大聚类范围，直到达到阈值。
  * 聚类范围：可变地址位的最大最小值 
  * 密度：种子地址在聚类范围种的比例

* 相比于entropy，能够发现更多的IPv6地址,且在较小集合上的地址探测命中率高于entropy/ IP
* 首次提出**IPv6生成地址的别名地址和别名前缀问题**
  * 很可能探测到的大部分存活地址都指向同一个接口
  * **精确检测别名的方法：指纹识别**
* 一种简洁的别名前缀发现方法【明显看效果不高，可以调研alias resolution进行别名消除】：
  * IDEA:**既然IPv6别名地址大量指向同一个接口，那么在别名前缀下随机生成一定数量的地址都被检测为存活，那么这些地址和可能就是别名地址，这个前缀就是别名前缀。在/64前缀下的IPv6地址空间如此庞大，随机生成的地址不可能都存活**
* 6Gen在时间性能方面: 随种子地址集合的增大迅速增大，呈**指数增长**。【**太慢！！！！！**】

## 6Tree：分裂聚类的IPv6地址探测技术

**Contribution：解决6Gen时间效率；基于反馈的别名去除办法**

* 核心步骤：分析已知种子地址，将其理解为高维向量，进行分层次聚类，已生成空间树的数据结构，该结构表征不同维度中的半字节值变化。
* Basic Assumption：**某些维度比其他维度具备更多可变值，扫描器可以先搜索这些维度，可能发现这些维度中的变化情况得到新的值**
* **根据扫描结果作为反馈可以动态调整生成的地址空间**【6Scan的强化学习也是干的这件事情】

* 优势：
  * 能够处理较大的种子地址输入
  * 探测到更多的地址：274万种子地址，生成3亿待测地址，得到636万存活地址，除去别名还剩167万。
* 99.5%的别名地址聚集在别名前缀下面

## In General

* 6Gen：时间效率太低
* Pattern based: 在随机化过多的情况下效率也较低
* 6Tree:经过实验测试，有反馈的情况下算法命中率优于其他模型
* 别名问题，实际也可以算是存活的地址，但是对于算法的存活地址可能产生影响
* 处理大量种子地址作为输入的能力
* 随着IPv6种子地址集合不断增大，整体存活地址必然表现出随机性，因此需要有一定的办法解决地址较为随机的情况。

## Aliased prefix detection 

**别名前缀的判别方式：**

* **假设：在极大地址空间的IPv6前缀中，随机探测的IPv6地址不可能都进行响应。**
* **因此：若在一个前缀中随机探测一定数量的IP地址超过一定阈值数量的地址进行了响应，那么该前缀可以认为是一个别名前缀**

### Multi-Level Aliased Prefix Detection 

* **Murdock的别名判别方式：向/96的prefix中随机发送3个探测包，全响应则表示很可能为alias prefix** 
  * 对于星链的user router，可能需要关注 /64区间。

   * 相比于之前的Murdock只关注/96区间的murdock,探测过程中同时关注了不同长度的prefix lengths【/64到/124,每次半字节一个单位】
     * 向目标网段中的伪随机地址发送16个包，**TCP/80  & ICMPv6**
     * 16个包都响应，则表示此网段为aliased prefix 
     * Table3中的例子：
       * 目标：2001:db8:407:8000::/64 
       * 在16个/68中随机生成一个IP地址，判断是否全部响应，如果全部响应则表示该/64前缀为alias，
     * **步长为4：正好半个字节，信息熵不多不少**

* **别名查询结果可能随前缀而变化：**
  * **/96被判定为非别名**
  * **其包含的16个/100可能其中9个被标注为别名**；
  *  若使用完全随机的方式，可能16个地址都落入9个别名空间，导致/96判定为别名，**【因此别名判定不能纯随机生成】**

* 考虑到丢包可能导致的“别名判断错误”：**多协议响应融合、多日期滑动窗口**组合判定被引入
  * 多协议：ICMPv6 和 TCP 80任意一个响应即可。
  * 滑动窗口：避免单次请求无响应带来的误差

#### iTTL[13,46,55] 

* **TTL value for response packages** 
  * 同一前缀甚至同一IP地址的TTL值未必保持恒定。不一致的原因：
    * 路由变更、TTL-manipulating middle box等
  * 因此，**replace the raw TTL metric with the likely initial TTL value chosen by a host**
  * As the iTTL can be only one of four values(32, 64, 128, or 255), we use differing iTTL as a negative indicator for APD: **many different iTTL values suggest a non-aliased prefix.**【iTTL越多，越可能不是别名前缀】
* 将已知类型的TTL计算iTTL后，作为一类特征，多个特征聚类后，很可能直接出现几个明显的峰值：64、128、255

**iTTL的设置长度分析**

将接收到的TTL值向上取整到最近的2的幂，作为推测的iTTL

* 因为IPv4/IPv6报文在发送时会设置一个初始TTL/Hop Limit,每经过一跳减1，所以探测者看到的是: TTL_received = iTTL - hopcount
  * 这样会导致不知道实际经过了多少跳

##### 常见的Initial TTL

绝大多数操作系统使用的固定值：

| 操作系统                         | iTTL    |
| -------------------------------- | ------- |
| Linux、macOS、Android、BSD       | 64      |
| Windows、Windows server          | 128     |
| Cisco、Juniper、HUAWEI、Mikrotik | 255     |
| **路由器设备**                   | **255** |

* 因此网络测量中通常假设：iTTL∈{32,64,128,255}
* 收到TTL=58 
  * iTTL  = 64    
  * hopcount = 64-58 = 6
* 为什么是向上取整到2的幂，因为历史上的TTL默认值经常设置成32、64、128、256约等于255【发出去就要少一跳】
* **Starlink不同设备可能使用不同的iTTL**

#### TCP SYN/ACK TTL

* 发送 TCP SYN，接收 SYN-ACK 或 RST,以类似的方式计算iTTL
* 有点：TCP更实用，避免ICMP被防火墙过滤

##### MSS(Maximum Segment Size)

* TCP SYN 中最大报文段长度，不同操作系统默认的MSS不同：

  | 操作系统      | Default TCP Windows |
  | ------------- | ------------------- |
  | Linux/Android | 1460                |
  | Windows       | 1460/1440           |
  | Cisco ISO     | 1460/1380           |

* **inconsistent MSS value also determine that two IP are not alias**

##### TCP Options 顺序

* TCP SYN 包中选项的顺序，常见选项：`MSS, Window Scale (WS), SACK Permitted, Timestamp (TS), NOP`, 不同os的默认顺序：

| 操作系统  | TCP Options         |
| --------- | ------------------- |
| Linux 5.x | MSS,SACK,TS,WS,NOP  |
| Windows   | MSS,NOP,WS,NOP,SACK |
| mac       | MSS,TS,SACK,WS,NOP  |

* 很多osFingerprint用此特征进行唯一识别，对中间设备NAT/负载均衡影响极小

#### TCP windows size 

* 操作系统TCP堆栈默认接收窗口大小, 不同操作系统TCP window 默认值不同

| 操作系统                 | Default TCP Windows  |
| ------------------------ | -------------------- |
| Linux                    | 29200 / 5840 / 64240 |
| Windows                  | 65535 / 8192         |
| BSD / macOS/Cisco/Router | 65535                |
